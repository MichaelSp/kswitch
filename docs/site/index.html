Features

Everything an operator needs

Designed for the reality of managing hundreds or thousands of Kubernetes clusters across multiple providers and locations.

🔍

Unified Fuzzy Search

Search all contexts across every provider and kubeconfig file in one instant fuzzy search — no more grepping dozens of files. Hot-reload adds new kubeconfigs on the fly.

🪟

Terminal Window Isolation

Each terminal window targets a different cluster independently. kswitch never modifies the original kubeconfig — a temporary copy is created per terminal session.

⏱️

History Navigation

Every {context, namespace} tuple is recorded. Jump back instantly with kswitch . or step through history with kswitch -.

🏷️

Context Aliases

Give human-friendly names to cryptic generated context names without touching the underlying kubeconfig. Aliases are local, reversible, and listable.

⚡

Search Index Cache

Local index for instant results even across massive directories or slow remote stores like Vault. Configure refresh intervals per-store for fine control.

🪝

Extensible Hooks

Run arbitrary executables or inline commands before search — sync kubeconfigs from Git, refresh tokens, rotate credentials. Fully composable with your toolchain.

🔐

Sanitized Preview

Live preview of the selected kubeconfig with credentials stripped, so you can verify the right cluster before switching. Disable it to reduce API traffic against Vault.

🌐

Namespace Switching

Change namespace with kswitch ns, powered by a self-updating local cache for instant search across all available namespaces per cluster.

🔁

kubectx Drop-in

Alias kubectx=switch and keep your muscle memory. kswitch is a compatible superset of kubectx, supporting all its basic usage patterns.

How it works

Simple by design

A thin shell wrapper and a Go binary. No daemons, no servers, no cloud sign-in required.

01

Source the shell function

Add source <(kswitch init zsh) to your .zshrc. This installs the switch function and tab completion automatically.

02

Type switch

kswitch discovers all configured kubeconfig stores and builds a combined list of context names from local files, Vault, cloud APIs, and more.

03

Fuzzy search & select

Type to filter by any part of the context name or parent directory. A sanitized preview pane shows the kubeconfig live. Press Enter to confirm.

04

Isolated context exported

kswitch writes a temporary copy of the kubeconfig and exports KUBECONFIG into your shell — never modifying the original file.

kswitch Documentation

kswitch is the single pane of glass for all of your kubeconfig files. Caters to operators of large scale Kubernetes installations and is designed as a drop-in replacement for kubectx.

Quick Start

After installation, simply type switch (or kswitch if you set the alias) in any terminal to open the fuzzy context switcher.

# Switch context interactively
$ switch

# Switch to last-used context
$ switch .

# Switch to previous context
$ switch -

# Change namespace interactively
$ switch ns

# List all contexts (with optional wildcard filter)
$ switch list-contexts "*-dev-?"

Key Concepts

  • Terminal isolation — each terminal window has its own copy of the active kubeconfig; switching in one window never affects another.
  • SwitchConfig — a YAML file at ~/.kube/switch-config.yaml that defines which kubeconfig stores to search.
  • Hooks — executables or inline commands run before search to keep kubeconfigs fresh.
  • Search index — optional local cache of context names for instant search, even against remote stores.

Difference to kubectx

kubectx is great for a handful of kubeconfigs but lacks support for large Kubernetes installations where clusters spin up on demand or are stored in various stores (e.g., Vault).

kswitch is built for a world where Kubernetes clusters are treated as cattle, not pets. It adds:

  • Unified search over multiple providers
  • Terminal window isolation (no shared current-context)
  • Context history navigation
  • Improved search for cryptic context names (parent directory shown)
  • Aliases without modifying kubeconfig files
  • Search index for large-scale installations
  • Hooks for custom integration

Installation

The kswitch installation consists of the kswitch binary and a shell function that must be sourced. Always invoke kswitch via the shell function — not the binary directly — so that KUBECONFIG is exported into your current shell session.

Option 1 — Homebrew

$ brew install kswitch

Option 2 — GitHub Release

# Replace OS and VERSION as needed
OS=linux   # linux | darwin
VERSION=v0.9.3
$ curl -L -o /usr/local/bin/kswitch \
    https://github.com/MichaelSp/kswitch/releases/download/${VERSION}/kswitch_${OS}_amd64
$ chmod +x /usr/local/bin/kswitch

Option 3 — From source

$ go install github.com/MichaelSp/kswitch@latest

Source the shell function

Zsh

$ echo 'source <(kswitch init zsh)' >> ~/.zshrc
$ echo 'alias s=switch'            >> ~/.zshrc      # optional short alias
$ echo 'alias kswitch=switch'      >> ~/.zshrc      # optional kswitch alias
$ source ~/.zshrc

Bash

$ echo 'source <(kswitch init bash)' >> ~/.bashrc
$ echo 'alias s=switch'             >> ~/.bashrc
$ source ~/.bashrc

Fish

$ echo 'kswitch init fish | source' >> ~/.config/fish/config.fish

PowerShell

> kswitch_windows_amd64.exe init powershell >> $PROFILE
> Register-ArgumentCompleter -CommandName 'kswitch' -ScriptBlock $__kswitchCompleterBlock >> $PROFILE
> . $PROFILE

Verify installation

Open a new terminal (or re-source your config file) and run switch (zsh/bash) or kswitch (fish/powershell). You should see the fuzzy context picker.

If you see "Error: you need to point kswitch to a kubeconfig file", set up your kubeconfig stores.

How it works

kswitch consists of two components working together:

  • switch.sh — a shell function (switch()) that is the user-facing entry point.
  • kswitch binary — a Go binary that handles context discovery, fuzzy search, and kubeconfig manipulation.

Execution flow

  1. User runs $ switch (the sourced shell function).
  2. The shell function discovers and executes the kswitch binary from $PATH.
  3. kswitch searches for kubeconfigs in the configured kubeconfig stores.
  4. kswitch displays a fuzzy search over all context names.
  5. The user selects a context.
  6. kswitch creates a temporary copy of the selected kubeconfig, sets current-context, and writes it to ~/.kube/switch_tmp.
  7. kswitch writes the path to the temporary kubeconfig to stdout.
  8. The shell function captures the path and runs export KUBECONFIG=<path> — making the change visible only to the current terminal.

This design means each terminal window operates on its own copy of the kubeconfig, achieving terminal window isolation without modifying the original file.

Kubeconfig Stores

kswitch can search over multiple directories, files, and Kubeconfig stores. Without configuration it defaults to ~/.kube/config.

The SwitchConfig file is expected at ~/.kube/switch-config.yaml or can be set via --config-path.

Supported stores

  • filesystem — local filesystem, recursive directory search
  • vault — Hashicorp Vault secret store
  • gardener — Gardener landscapes
  • rancher — Rancher clusters
  • EKS, GKE, AKS, DOKS, Exoscale, OVH, Akamai/Linode, CAPI, and more

Basic filesystem config

# ~/.kube/switch-config.yaml
kind: SwitchConfig
version: v1alpha1
kubeconfigStores:
  - kind: filesystem
    kubeconfigName: "config*"
    paths:
      - ~/.kube

Combined multi-store search

kind: SwitchConfig
version: v1alpha1
kubeconfigStores:
  - kind: filesystem
    paths:
      - ~/.kube/static-kubeconfigs/
  - kind: vault
    paths:
      - "path/in/vault"

Optional stores

Mark a store as required: false to suppress errors when it is unreachable (e.g., a VPN-only cluster not always accessible).

kind: SwitchConfig
kubeconfigStores:
  - kind: filesystem
    required: false
    paths:
      - ~/vpn-clusters/

Disable context name prefix

By default each store prepends a prefix (e.g., the parent directory). Disable it with showPrefix: false.

kind: SwitchConfig
kubeconfigStores:
  - kind: filesystem
    showPrefix: false
    paths:
      - ~/.kube/static-kubeconfigs/

Search Index

The search index is a local file that maps context names to their kubeconfig paths, allowing kswitch to bypass querying the store on each invocation. Especially useful for:

  • Large directories where filesystem traversal is slow
  • Remote stores (Vault, Gardener) to reduce API requests and latency

Index files are stored at ~/.kube/switch-state/switch.<store>.<id>.index.

Enable for all stores

The refreshIndexAfter field sets the TTL for the index of every configured store.

kind: SwitchConfig
refreshIndexAfter: 1h
kubeconfigStores:
  - kind: filesystem
    paths:
      - ~/.kube/my-kubeconfigs/

Enable per store

kind: SwitchConfig
kubeconfigStores:
  - kind: filesystem
    id: unique-1
    refreshIndexAfter: 3h
    paths:
      - ~/.kube/static-kubeconfigs/

Skip index for one request

$ switch --no-index

Kubeconfig Cache

For remote kubeconfig stores, kswitch downloads the kubeconfig file on each invocation. The cache stores a local copy so subsequent invocations are faster and require fewer API requests.

Note: the search index and the kubeconfig cache are independent. Configure both for best performance with remote stores.

Enable filesystem cache

kind: SwitchConfig
version: v1alpha1
refreshIndexAfter: 10h
kubeconfigStores:
  - kind: vault
    id: example
      # ... vault config ...
    cache:
      kind: filesystem
      config:
        path: ~/kubetest/cache

Cache files are named by content hash and store kind, e.g. 08df4a6d.vault.example.cache. It is safe to point multiple stores at the same directory.

Note: cache files are not encrypted. Protect the directory with appropriate filesystem permissions.

Clean the cache

$ switch clean

Cleaned 3 files from temporary kubeconfig directory.
Cleaned 15 files of vault.example cache

Hooks

Hooks allow kswitch to call arbitrary executables or inline shell commands before the fuzzy search opens. Think of them like Git pre-commit hooks — composable scripts that keep your kubeconfigs up to date.

List configured hooks

$ switch hooks ls

+---------------------------+------------+----------+----------------+
| NAME                      | TYPE       | INTERVAL | NEXT EXECUTION |
+---------------------------+------------+----------+----------------+
| sync-dev-landscape        | Executable | 6h0m0s   | 4h39m0s        |
| sync-live-landscape       | Executable | 24h0m0s  | 3h47m0s        |
+---------------------------+------------+----------+----------------+

Run a hook manually

$ switch hooks --hook-name=sync-dev-landscape

Hook calling an executable

kind: SwitchConfig
hooks:
  - name: sync-dev-landscape
    type: Executable
    path: ~/bin/gardener-landscape-sync
    arguments:
      - "sync"
      - "--landscape-name"
      - "dev"
    execution:
      interval: 6h

Inline command hook

kind: SwitchConfig
hooks:
  - name: garbage-collect
    type: InlineCommand
    execution:
      interval: 6h
    arguments:
      - "switch clean && echo 'GC complete.'"

Hook execution state is stored in ~/.kube/switch-state (configurable with --state-directory).

Command Completion

Command completion is included automatically when you source the shell function via kswitch init <shell>. The manual steps below are only needed if you want to install completion separately.

Zsh

$ echo 'source <(switch completion zsh)' >> ~/.zshrc

Bash

$ echo 'source <(switch completion bash)' >> ~/.bashrc

Fish

$ echo 'kswitch completion fish | source' >> ~/.config/fish/config.fish

PowerShell

> echo 'kswitch completion powershell'                                         >> $PROFILE
> Register-ArgumentCompleter -CommandName 'kswitch' -ScriptBlock $__kswitchCompleterBlock >> $PROFILE
> . $PROFILE

Aliases

Aliases let you assign a human-friendly name to any context without modifying the underlying kubeconfig file. Aliases are local to your machine and can be removed at any time.

Create an alias

# alias NAME=FULL_CONTEXT_NAME
$ switch mediathekview=gke_mediathekviewmobile-real_europe-west1-c_mediathekviewmobile

# alias for the current context
$ switch alias prod-us=.

List aliases

$ switch alias ls

+---------------+-------------------------------------------------------------------------------+
| ALIAS         | CONTEXT                                                                       |
+---------------+-------------------------------------------------------------------------------+
| mediathekview | mediathekview/gke_mediathekviewmobile-real_europe-west1-c_mediathekviewmobile |
+---------------+-------------------------------------------------------------------------------+
| TOTAL         | 1                                                                             |
+---------------+-------------------------------------------------------------------------------+

Remove an alias

$ switch alias rm mediathekview

Execute Commands

Use wildcard search to run any command against a set of matching clusters simultaneously. Powerful for quickly running a command across a landscape and comparing output.

Basic exec

# Run against all clusters matching "*-dev-?"
$ switch exec "*-dev-?" -- kubectl get ns

Run a script

$ switch exec "*-dev-?" -- bash ./check-health.sh

Inline shell commands

Specify a default shell in your SwitchConfig, then use inline commands:

# ~/.kube/switch-config.yaml
kind: SwitchConfig
execShell: "/bin/bash"
$ switch exec "*-dev-?" -- 'for i in 1 2 3; do echo "cluster: $i"; done'

Wildcard syntax

  • ? — matches exactly one character
  • * — matches any number of characters (including zero)
$ switch list-contexts "*-prod-eu*"
